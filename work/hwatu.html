hwatu · systems / agent tooling · 2026

AI agents write frontend code they never see.

hwatu is a daemon-based WebKit browser where opening, driving, and closing a real rendered window costs about as much as running ls. It turns visual verification from a once-at-the-end tax into something an agent does after every edit.

The problem

The dominant failure mode of AI frontend work isn’t bad code generation, it’s a broken feedback loop. The agent edits a component, declares “done,” and never looks at the pixels. When it does try to look, the standard tool is a fresh headless Chromium: seconds of cold start and hundreds of megabytes per context. At that price, verification happens once at the end, if at all.

Make the middle step effectively free and the loop changes shape: checking your work stops being a decision and becomes a reflex.

The idea

Browsers conflate two things: the engine (slow to start, RAM-hungry) and the window (what you actually asked for). hwatu splits them the way emacsclient and wezterm do. A daemon, hwatud, owns WebKitGTK 6 and a prewarmed pool of WebViews. The client, hwatu, is one Unix-socket roundtrip.

Command to mapped, loading window on a warm daemon. 20 runs per mode, budget-gated in CI. Each extra window costs ~56 MB PSS over the daemon floor, one shared engine instead of one Chromium per context.

Full WebKit rendering, not a half-engine: layout, CSS, WebGL, media. Screenshots show what a user would see. Fast-but-blind DOM automation misses exactly the visual bugs verification exists to catch.

Driving it

The protocol is one newline-delimited JSON request per socket connection. No tool schema, no session objects, no WebSocket, cheap for token budgets, trivial from any language. The CLI is the same protocol with argv parsing: 

$ hwatu localhost:3000
3                # window id; headless under an agent
$ hwatu screenshot 3 /tmp/check.png
$ hwatu eval 3 'document.querySelector(".price").textContent'
"$1,299"
$ hwatu console 3
[error] Uncaught TypeError: cart is undefined  app.js:41
$ hwatu close 3

Snapshot, screenshot, eval, click, type, scroll, upload, console: the surface an agent needs to open a page, interrogate it, and move on. When the CLI detects an agent environment (CLAUDECODE, CURSOR_AGENT, …) it defaults to headless, so a forgotten flag never drops a window into the human’s window manager. And any headless window is a live session: hwatu focus 3 promotes it into the user’s tiling WM so the human can watch or take over.

Design decisions

  • No tabs.A tab is a window. The tiling WM, or the agent harness, is the tab manager. One abstraction deleted, two managers unified.
  • Headed and headless are window properties, not launch modes.The same session can be invisible for the agent, then focused for the human. No relaunch, no lost state.
  • Content blocking compiled into WebKit’s network process.EasyList evaluated natively: zero JS, zero UI, zero spawn cost. Verification pages load like a user with an adblocker sees them.
  • Per-socket daemons and sessions.Each agent harness can own an isolated daemon; crash snapshots are scoped to the socket that owns them.
  • Every number is measured.The spawn benchmark fails CI if the median regresses past budget. Performance claims are tests, not marketing.
