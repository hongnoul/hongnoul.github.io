A desktop built around agents, not apps.

Most desktops are organized around a human clicking between applications. Mine is organized around coding agents doing the work while I direct: niri gives every agent its own column on an infinite strip, jcode is the primary interface to the machine, and hwatu makes looking at the result free.

The window manager

niri is a scrollable-tiling Wayland compositor. Instead of cramming windows into a fixed grid or juggling numbered workspaces, the workspace is an infinite horizontal strip and the screen is a viewport onto it. New windows never resize existing ones, they just extend the strip.

I didn’t start here. The previous setup was Hyprland with a 2×2 window split across two workspaces: eight windows, hard ceiling. Every new agent meant evicting an old one or spawning another workspace to forget about. That cap quietly set the size of the fleet.

That model happens to be exactly right for agent-heavy work. A grid asks “where do I fit this?” every time something opens. A strip just grows. Navigation is vim keybinds throughout: alt+h and alt+l walk the strip the way h and l walk a line. Each running agent keeps a full, undisturbed terminal column, and adding a fifth agent costs nothing but width. Drag the viewport:

The agent

jcode is the interface I actually live in. Not an editor with a chat panel bolted on, a coding agent that owns the terminal. It writes its own launch hotkeys into niri’s config, so starting a session anywhere is one chord:

chordopens
alt+;jcode in ~, the “anything” session
alt+'jcode in the current main project
alt+shift+'jcode in self-dev mode, on its own source
alt+`jump to the oldest agent waiting for input
alt+shift+`triage: cycle every open session

The bindings live between managed markers in config.kdl - jcode rewrites that block itself when its build or hotkeys change.

The pattern that falls out: I don’t context-switch between projects, I visit agents. alt+` is the closest thing to an inbox: it jumps the viewport to whichever column is blocked on me, oldest first. The strip holds the state; I supply judgment and move on.

The self-dev session deserves a note: jcode routinely modifies its own harness: hotkeys, tools, the compositor config it runs under. My ~/.config/niri/ has a dozen timestamped config.kdl.bak-jcode-hotkeys-* files, which is what it looks like when the agent edits its own environment and leaves an undo trail.

The bar

The waybar workspace ticker is where the fleet becomes legible. The stock module numbers workspaces and stops there; this one is a custom Rust binary that subscribes to niri’s event stream, walks each window’s /proc cmdline to find the ones running jcode, and cross-references them against ~/.jcode/session_status/ - the per-session state files every agent keeps updated as it works.

The addressing is a spreadsheet. Workspaces are letters, columns are numbers, and the focused window reads as a cell reference: A03 is workspace A, fourth column; hop down a workspace and you’re in B00. It turns “where was that agent?” into coordinates - an infinite strip per row, rows stacked alphabetically, exactly the mental model a grid of running sessions deserves. The bar renders the whole sheet as one line: a tick per window, the focused cell’s address highlighted, and every tick color-coded by what its agent is doing right now:

colormeaning
■ greenagent is working - leave it alone
■ yellowwaiting on my input - this is the queue
■ redstalled or dead - the session needs rescuing
■ dimfresh or idle - spawned but not yet tasked

With 12+ sessions live, this is the difference between managing a fleet and losing one. A glance at the bar answers “who needs me” without visiting a single column, and it pairs directly with alt+`: the bar says someone is blocked, the chord takes me to them. The status files update event-driven, not polled, so a session flipping from working to waiting shows in the bar the same instant the agent stops.

Credit split on this one: the ticker itself is Jeremy Huang’s design, part of the jcode setup this whole rice descends from. The session status color coding is the new addition, patched in on this machine - by a jcode agent editing the bar that reports on jcode agents, which felt appropriately circular.

Verification

The third leg is hwatu, the 13 ms daemon browser. It closes the loop the other two open: agents produce frontend changes constantly, so seeing the rendered result has to cost nothing. Two niri window rules do the etiquette: 

// hwatu --background: agent-verification windows must not steal focus.
window-rule {
    match app-id="hwatu-background"
    open-focused false
}

An agent can open, screenshot, and close real rendered pages in a hot loop without a single window flashing in front of me. When I want to see what it sees, hwatu focus promotes the invisible session into a normal column on the strip.

Why this shape

  • Columns, not workspaces.Workspaces are places you have to remember. A strip is a sequence you scroll. With 4-6 agents running, spatial order (oldest work leftward) beats naming things.
  • The WM is the tab manager.hwatu has no tabs and jcode sessions have no session switcher. Every unit of work is a window, and niri is the single manager for all of them. One abstraction, everywhere.
  • Agents get first-class citizenship.Window rules, focus etiquette, and launch hotkeys are designed for processes that open windows while I'm not looking. The human default (steal focus, animate, notify) is wrong when the “user” is a daemon.
  • Config the agent can edit.niri's KDL config is plain text with obvious structure. That's not just nice for me. It means jcode can safely patch its own hotkeys, with backups, without a settings UI in the way.
  • Everything keyboard, nothing modal.No dock, no alt-tab overlay, no drag-to-tile. Vim keybinds everywhere: h/j/k/l move focus in niri, the same motions drive the editor. The hands stay on the row where the prompts are typed.
