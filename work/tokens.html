Cache reads dominate because agentic loops replay long contexts on every provider call. “Fresh” tokens (input + output) are the smaller, pricier slice.

How it works

  • Source of truth: jcode session files. Every jcode session persists its per-message token_usage (input, output, cache read, cache write). An exporter sums them into local-date buckets.
  • Static publishing, no backend. The exporter writes tokens.json (a few hundred bytes: dates and counts only, never session content) and a systemd user timer commits and pushes it daily. This page is plain HTML that fetches the blob and draws SVG.
  • Same pipeline as the desktop widget. My waybar module tails jcode logs for a live tok/s burn rate and scans the same session files for daily totals. This page is the public, historical view of that number.

Data updates daily via a systemd timer on the machine doing the burning. Counts include cache reads, which is why the numbers look like a national debt ticker.
